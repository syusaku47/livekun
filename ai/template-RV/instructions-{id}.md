## 指示

### 指示書

- この文書が指示書です
- ファイル名は `instructions-{id}` となっているので `{id}` はファイル名から取得してください
- タスクが全て完了した際は ../000 を実施してください
- created-at, updated-at は YYYY-MM-DD hh:mm:ss の形式で記述してください
- created-at, updated-at は日本時間を取得し記述してください

### 計画書

- 計画書は `./plan-{id}` に記載してください
- 記載する際は obsidian の記法で記述してください
- `plan-{id}` というファイルがテンプレートなので、テンプレートをもとに記載してください

### 報告書

- 報告書は `./report-{id}` に記載してください
- 記載する際は obsidian の記法で記述してください
- `report-{id}` というファイルがテンプレートなので、テンプレートをもとに記載してください
- Markdown のテーブルを用いた方が読みやすいと判断できる場合はテーブルを用いて記述する
- Mermaid 記法を用いて視覚的に表現できる箇所がある場合は Mermaid 記法を積極的に用いてください
	- Mermaid 記法を用いる場合は予約後の重複などのエラーが起きないように注意してください

### タスク

- [ ] ブランチ名を取得する (FI-NNNN は機能毎に振られたIDのため、仕様を確認する際に用いる)
- [ ] ブランチ名から機能IDを取得する (FI-NNNN : N = 0  ~ 9 の整数)
- [ ] 機能ID をもとに 機能定義書 を確認し必要な機能を把握する
- [ ] API仕様書から関連するAPIの仕様を把握する
- [ ] 機能ID をもとにテスト実行結果を把握する
- [ ] 関連箇所の 旧ソース を確認する
- [ ] 機能ID をもとにテストケースを把握する
- [ ] 機能定義書に対してテストケースのカバレッジが充分かを検証する
	- [ ] 不足があれば必要なケースの提案も行う
- [ ] ソースコードに対してテストケースのカバレッジが充分かを検証する
	- [ ] 不足があれば必要なケースの提案も行う

- [ ] 指摘内容が間違いないかを確認する
	- [ ] 必要な実装がすでに行われている可能性を該当箇所を使用してる別のソースを確認してチェック
	- [ ] 該当箇所が依存している別のソースを確認してチェック
	- [ ] 例外がそもそも発生しない作りにしていないかをチェック
- [ ] 指摘内容が間違っていた場合はレビュー内容を修正する

## 制約

- このレポジトリ内のコードを直接編集することは禁止です
- このブランチが作られて以降、このブランチでコミットされた変更のみを対象としてください
- このブランチが作られる以前に実装された内容については言及しないでください
- 指摘箇所はわかりやすいように必ずファイル名 (パス) と行数を書くこと
- 指摘箇所の該当コードを抜き出し記載すること
- 指摘箇所に対しては具体的な修正方法の提案も行うこと
- 指摘箇所は参考にした仕様書/旧ソース/テストコードの具体的な場所も記載すること
- レビュー観点としては以下を参考にする
	- 機能の実装もれがないか
	- コーディングスタイルが統一されているか
	- 設計書通りに動くか
	- APIは仕様通りに使えているか
	- 可読性は高いか
	- コーディングの基本に忠実か
		- 命名規則に忠実であるか
		- 変数•関数のスコープは最小限にする
		- 長すぎるロジックは分割する
		- 共通のロジックは共通のロジックとして切り出す
		- ディレクトリ構成に従いロジックをかく
		- コメントを入れる
	- パフォーマンス問題ないか
		- React の無駄なレンダリングがないか
		- React のベストプラクティスな実装になっているか
		- 適切な状態管理が行えているか
